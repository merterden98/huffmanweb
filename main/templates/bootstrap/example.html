With the advent of larger data, compression plays an ever increasing role in society. A majority of modern day compression algorithms rely on the same principles layed out decades ago. Specifically lossless compression schemes have been limited by a lower bound on entropy and are thus optimal with regards to binary storage. Huffman Coding is one such algorithm. The core of Huffman Coding is reliant on heap structures, and as such is limited by the comparison models lower bound. Although I will make no attempts in lowering the entropy bound for compression, I will demonstrate speeding the Huffman Coding process with the usage of integer data structures.
How can we measure the amount of information that comes from a source? Information theory is a broad section of computer science that attempts to answer that question. Although we won't dive into the specifics of information theory, having a simple understanding of information and entropy proves itself to be very useful in the discussion of compression. Information is defined as $$I(x) = \log\frac{1}{P(X = x)}$$ Although not intuitive at first as to why information is defined that way, a few examples will quickly show the selection of the logarithmic function.

Firstly if an event has a low probability of happening it should be able to convey more information otherwise we would expect it to occur!, $$\log\frac{1}{0.001} > \log\frac{1}{0.5}$$ Secondly information is additive, if two events happen given they are independent we should learn no less information from either event. This is from the simple log addition property. $$\log\frac{10}{7} + \log7 = \log{10}$$ Using the definition of information we define entropy. Entropy is the weighted average of information, or in other terms the expected amounted of information that we can get from a source. $$H(S) = \sum_{i=1}^{n} p(i) * I(i) = -\sum_{i=1}^{n} p(i) * log(p(i))$$
Since we're working with computers that operate off of binary, we define our alphabet A to have letters A = {0, 1}. The smallest amount of addressable information on modern computers is a byte (8-bits) and thus in order to represent the English alphabet we need to use an entire byte per character. For instance the letter A on a computer is '01000001', b is '01000010' etc... To represent 26 letters; however, we really don't need all 8 bits. In fact we can pack all bits we need into a byte, and interpret them as we read them in.


The issue arises when we attempt to assign bits arbitrarily to the english alphabet. Lets say that we're trying to compress the characters {a, b, ' '} the last character being a space. We give them the bit values {0, 01, 010} respectively. For our sample encoding let's try deciphering the bitstring '00100010010'. If a computer can only read from left to right is the code above 'abaa ba' or 'a a ba' or any other possible combination? Ambiguity would probably be very bad if you wanted to decipher valuable information. The above problem arises from the fact that the letters share the same prefixes, like b and ' ' both start with the coding for the letter a.

The solution for the problem is through writing codes that do not share the same prefixes. Let's encode the above alphabet again but instead use { 0, 10, 110 }. Each character in this case does not share a prefix with the last. Deciphering the bitstring '011010010' is far easier. Starting from the left, this code deciphers to, 'a bab'. The core of Huffman Coding is reliant on the notion of uniquely decodable prefixes.
The last prerequisite for Huffman Coding is understanding the optimality of compression. Specifically what is the smallest number of bits I need in order to retain all the information I need? We will show this using Entropy and Kraft-McMillan's inequality. Kraft-McMillan's inequality is defined as: $$\sum_{i=1}^{n} 2^{-l_i} \leq 1$$ Given our source alphabet S, l is the length of the prefix code of a specific letter, i,e if a = '001', l = 3. Kraft-McMillan states that if I have uniquely decodeably prefix codes, the sum 2 raised to the negative of their lengths is always less than or equal to one. The proof for this is extensive and will be included in the resources. The key word to pay attention to in the inequality is uniquely decodable, otherwise after compression we would not be able to decode our text. Optimal compression bounds are said to be: $$H(S) \leq \overline{l} \leq H(S) + 1$$ Where H(s) is entropy and l bar is the average length of the codes. A probabilistic way of getting the average length of code given alphabet: $$A = {a_1, a_2 ... a_n}$$ and letter frequencies: $$ P(A) = {p(a_1), p(a_2) ... p(a_n)}$$ is: $$\overline{l} = \sum_{i=1}^{n} p(a_i)l_i $$ The difference between the source entropy and the average code length is: $$H(S) - \overline{l} = -\sum_{i=1}^{n} p(a_i) * log(p(a_i)) - \sum_{i=1}^{n} p(a_i)l_i$$ From log properties: $$H(S) - \overline{l} = \sum_{i=1}^{n} p(a_i) log(\lbrack\frac{1}{p(a_i)}\rbrack - l_i)$$ $$H(S) - \overline{l} = \sum_{i=1}^{n} p(a_i) log([\frac{2^{-l_i}}{p(a_i)}])$$ Since log is a convex function we can use Jensen's inequality to conclude that: $$H(S) - \overline{l} \leq log(\sum_{i=1}^{n} 2^{-l_i}) $$ From Kraft-McMillan's Inequality this results in: $$H(S) - \overline{l} \leq log(1) $$ $$H(S) \leq \overline{l}$$ Proving the lower bound for optimal code. For the lower bound proof we look at the same probability model and utilize the information function we defined earlier. We let: $$l_i = \lceil I(p(a_i)) \rceil = \lceil \log(\frac{1}{p(a_i)}) \rceil$$ The cieling function implies a lower an upper bound for our values. Specifically: $$I(p(a_i)) \leq l_i < I(p(a_i)) + 1 == \log\frac{1}{p(a_i)} \leq l_i < \log\frac{1}{p(a_i)} + 1$$ As the ceiling function can round up to the nearest integer.
Using the probability model, $$\overline{l} = \sum_{i=1}^{n} p(a_i)l_i $$ We now show that through substituting the upper bound on length, $$\overline{l} < \sum_{i=1}^{n} p(a_i) (\log\frac{1}{p(a_i)} + 1) $$ The above simplifies to: $$\overline{l} < H(S) + 1$$ We have therefore seen that if we compress code such that is is lossless it must be bound by $$H(S) \leq \overline{l} < H(S) + 1$$
The Huffman Coding algorithm is fairly simple. Given a file with the text "Hello World" we start with counting the frequency of the letters. Afterwards we sort the letters by the frequency in which they appeared in. From here we can construct a tree to handle encoding. The method of tree construction; however, is more unconventional as we build a Huffman Tree from bottom up. We select the letters with the lowest frequencies and construct a dummy parent whose left child is one letter and the right child is another. Place the newly constructed parent back into the list. Repeat until only one element is left. After a tree is constructed you can let every branch that goes to the left be a zero and every branch that goes to the right be a 1. These branches should then give you your encoding scheme.


Scanning through the original text an using the newly made code scheme, you can compress the letters. The caveat of Huffman Coding is that the constructed Tree needs to be saved, otherwise the data cannot be reinterpreted. To decompress a file we follow the bits in the previously constructed Huffman Tree until we hit a child node, we then place the childs value into the decompressed file.


Ultimately the limiting factor of Huffman Coding results from finding the nodes of the least frequency and placing them back into the list and resorting, which could take at most O(nmlgn) if we were to keep a list and had to keep resorting m times. A simple solution to this problem would be to use a structure that supported the following operations. Find min, insert, delete and successor. We would extract the lowest two frequency letters, combine them and insert them back into the structure. A fairly elementary data-structure that allows for these operations is a simple binary heap/priority queue, which would speed the Huffman construction to O(nlgn); however, would it be possible to speed compression even further? Convential comparison sorting bounds would say otherwise. The bounds defined by the comparison model are useful in regards to real numbers, but frequencies are guaranteed to be integers. Perhaps it is possible to exploit the fact that we are working with integers to speed compression.
Van Emde Boas Trees unlike most convential data structures are defined recursively and rely on bit properties to function. Before describing the layout of a van Emde Boas Tree it is easier to understand what a bitvector is. A simple way to think of one is imagining an array of booleans where each index position represents a true or a false. The bitvector on the right for instance shows that 1 and 6 exist. We can do this check in constant time by just checking the array index; however, asking the question what's the successor of 1 could take an entire scan through the array. Similarly, min and max queries can take at most O(n) time. We will attempt to incrementally solve this problem.
Our first attempt will be to block out the bit vector. In other terms we can keep a summary structure above that sections the bitvector into groups of size B. This summary structure will store whether or not values exist in the range of that block. Our new search becomes, check the summary structure, if what we are looking for exists (i,e minimum) then check the sub structure through a linear scan. This new time complexity becomes O(B + n/B) and is optimized when B is √n. The only problem that arises here is that if we delete a value we have to check that the block we are in is not empty.
In order to simplify mathematical calculations we define a function High(x) = floor( x / √n ) and Low(x) = x mod √ n To search for values we can now call High(x) to see if the summary vector contains our value, and then call Low(x) on the substructure. This is done in order to recursively define our structure. For instance if we were to insert(x), insertion now divides into a recursive insert into the summary structure and into the main bitvector structure. Finding min() for instance turns into first recursively finding min the summary structure, then finding the min on the main structure given the summarys min result. Finding max() is symmetric to min.
Successor on a bitvector is done be finding max() in the High(x) of that bitvector if it exists and is greater than x, successor is called on the Low(x) of that bitvector. Otherwise we are in the wrong block, so we call for successor to be found in the summary structure and return the minimum of those.
Currently we have a structure with a summary vector and a bitvector. However we can speed this structure up using a little more recursion. We define our recursive structure as follows. If our universe size is O(1) size just use a regular bitvector, otherwise divide structure in √n clusters with a summary structure of size √ n that keeps track of the clusters.


Since we have defined this structure recursively, lets analyze some operations. For instance take the search(x) operation. Searching for a value within our structure involves looking if High(x) cluster exists and if it does searching for Low(x) within that cluster. This operation requires a constant check followed by a recusive call into a subproblem of size √ n. The reccurence is modelled as $$ T(n) \leq T(\sqrt{n}) + O(1)$$ We solve this reccurence by, letting: $$ n = 2^m$$ Our reccurence becomes $$T(2^m) \leq T(2^{m/2)} + O(1)$$ Let: $$S(m) = T(2^m)$$ $$S(m) = S(m/2) + O(1)$$ We can solve S(m) using the master method, which solves to $$O(lgm)$$ For n this implies a time complexity of $$O(lglgn)$$ Which is great! However, if we look at operations like min() or max() we will encounter some issues. For instance for min() we not only have to recurse on the larger structure, but we also have to take into account the summary structure. This makes the reccurence for searching for min or max, $$T(n) = 2T(\sqrt{n}) + O(1)$$ which in when solved has the time complexity O(logn). In fact analyzing the other operations such as successor, delete etc... will result in a similar cases.
A better way to think about the the structure we have now, is by thinking of a well balanced BST. In terms of capabilities we have a BST that has a faster search capability O(loglogn). A good technique to speed up operations is by looking at what makes them slow. This technique can be applied to many other data structures and will be used later as well. We start with looking at min() and max(). We do not need to always search for min explicity each time. In fact a very simple method to not always search for these values is just by explicity storing them within the structure. So in other terms, have sections for both min and max. If we insert as normal and keep track of the new item, we can update min and max as we insert. In the end by adding a min and max field we make the time to find min and max O(1), although it may not seem like it, we also have sped up more processes.
Normal insertion on a surface level analysis requires two recursive calls, one on the summary and the other on the clusters. In other terms, if your summary structure is not empty at High(x), just insert into Low(x) in the clusters recursively, otherwise insert into High(x) in the summary and Low(x) in the clusters. The reccurence for this is $$T(n) = 2T(\sqrt{n}) + O(1)$$ which when evaluated results in $$O(log u)$$ In reality; however, we don't really have two recursive calls. Firstly because we store min and max values, and given an empty tree, insertion takes at most O(1) time as we just update the min and max field. Secondly we have to observe what the summary structure actually entails. If the summary structure is empty that implies that in the next step of the recursion we are guaranteed an empty structure otherwise it would not have been empty. Otherwise we just call insert onto our cluster using Low(x). This implies the only time we recurse into a summary cluster, we do constant work. With proper analysis the recurrence really is $$T(n) = T(\sqrt{n}) + O(1)$$

Deletion is symmetric to insertion. When we delete an element we remove Low(x) from the High(x) cluster recursively, afterwards we check if deleting the element resulted in the structure being empty and if so recursing into the summary structure. Adding a min and a max field implies that these fields need to be updated. Analyzing the recursive structure shows that we do not really recurse twice. If we delete the last element within a cluster all we need to do is recurse on the summary structure, otherwise we just recurse on the clusters as the summary structures do not need to be updated. This again becomes a O(loglogn) operation.
The last operation to understand is that of finding the successor. Other than handling the base case there are two cases to observe in finding a successor. The first case is if we are at a cluster and the value we are searching for is less than that clusters maximum, this implies that the successor of x has to be at least the max of the cluster. Otherwise we have to look into the summary array to find the successor of the current cluster and then return the min from the successor cluster. The reccurence for this is fairly simple and is $$T(n) = T(\sqrt{n}) + O(1) $$ which evaluates to $$O(loglogn)$$
x-Fast tries are an other integer data structure first proposed by Dan Willard in 1982. They primarily involve the manipulation of bit properties for their functionality. Foundations of this datastructure rely on understanding the benefits of what hash tables, tries and linked lists provide when in unison and thus keeping track of many, many, many pointers... The way an x-Fast is layed out is as a bit trie, where th left child is a 0, and the right child is a 1, much like how a Huffman Tree is shaped. In the case of an x-Fast there is a key distinction between internal and leaf nodes is all the actual stored values are stored in the leaves, the internal nodes serve as intermediate bit values.

With tries we have the properties of of a regular BST, in the sense that query, insert, delete, and successor searches all cost: $$O(logn)$$ To add to this property each layer of the trie also stores a hash table for the nodes. An immediate effect of this the ability to do constant time (with high probability), queries at the leaf level. Another subtle side effect this modification is the ability to binary searches on each level. A simple way to think of this is in binary '100' (4) and '101' (5) are different, but only in the last bit. In other words they share a common prefix. Although not immediately useful, binary searching the height is where we will shave off a log factor. Binary Searching: $$O(loglogn)$$ The second to last modification is converting the nodes on the leaf layer to be a linked list. The effect this has on the current data structure is if a query exists (an operation that takes O(1) time), the successor of that will also take O(1) as it is just the node to the right.
The last underlying problem is if a query does not exist and we need to find a successor. Speeding up this operation is the last addition needed to make the data structure into a x-Fast trie. The tools we already have are binary searching and this is where we can utilize the prefix property. During the construction of the x-Fast trie we also have extra conditions for internal nodes that have no left or right child. If an internal node were to have no left child it stores a descendant pointer to the smallest leaf in its right sub tree, and if an internal node were to have no right child it stores a pointer to the max leaf in its left subtree. Overall this means that we can binary search for the longest prefix and follow a descendant pointer to the next available leaf node. From there we travel either left or right on the leaf level, depending on either successor or predecessor search in O(1) time. In the end successor queries are: $$O(loglogn)$$
A simple recap of the successor search is check if the query exists in hash table on the leaf layer. If it exists we go to the node on the right of leaf node. If the node does not exist, we binary search on the layers of the tree to find the internal node with the longest prefix, afterwards we follow the descendant pointer and go to the right of the descendant. This operation is mainly limited by the binary search, and thus is $$\Theta(loglogn)$$

Successor and predecessor searches may be fast; however, we are limited by the insertion and deletion using tries. The comparison lower bound limits the insertion and deletion processes to $$\Omega(logn)$$ Inserting first requires finding a successor and predecessor, we can afford the time for this. The reason for this is that the node that will be inserted will be sandwiched between the successor and predecessor. After inserting the leaf we go from root to the inserted leaf, adding internal nodes and descendant pointers when necessary ultimately connecting the leaf to the root. $$\Theta(logn)$$ Deleting is fairly similar to insertion; however, this time we go about deleting internal nodes and reconnecting leaves such that they maintain predecessor and successor properties. Similarly this is $$\Theta(logn)$$
In the 1982 Willard's paper regarding the lower bounds of the predecessor problem, x-Fast tries were used as a precursor to the main structure of question, y-Fast tries. As seen previously, x-Fast tries are fairly good in terms of doing successor and predecessor queries; however, the moment successive insert operations come in, they are bounded by the lower bound $$\Omega(logn)$$

y-Fast, are an incremental improvement on the x-Fast trie, without that much extra requirements. The beauty of a y-Fast is that it ties together a technique we saw in speeding up the van Emde Boas Tree. The technique in question is using clustering and summary structures to divide the work. We first partition our keys and values into groups of size O(logn) (assume the data is sorted, we will prove the dynamic case later), among these groups each group has a representitive max value which will be stored in a summary x-Fast structure above.
To understand why we made groups of size log(n), it is easier to go through how the query function works. Searching for a value x first has us searching for the successor(x-1) within the x-Fast structure. As previously described this takes O(loglogn) time to do and will actually tell us which subgroup our query x will be in, as we can reverse engineer the group index if we store the max values within the groups. $$group(x) = \lceil\frac{successor(x-1)}{n}\rceil$$ After finding the correct subgroup, we can go through the binary search tree below normally and assuming it's balanced we can binary search within the subgroup for our query. As the subgroups contain O(logn) many elements our query will take O(loglogn).

The remaining operations to discuss follow the same pattern of finding a successor within the x-Fast structure and working on the subgroups. I will refer to the group finding that does the successor query as $$group(x)$$ The y-Fast insert, delete and successor functions are practically BST operations with an initial group query, and thus do not need much discussion; however, it should be noted they all are reduced to time O(loglogn) as a result of grouping.
Until now we have discussed the static problem, or in other terms we assumed that the input was nicely divided into perfect groups of logn to work with. The dynamic problem requires amortized analysis to properly understand. If we recall inserting into an x-Fast trie takes O(logn) to do, which is a limits the speed of our operations. An observation to make; however, is we do not need to always to insert into the x-Fast trie. In fact we only need to insert once per group representitive. To make the problem dynamic we start off with an empty structure, and insert the first element, as we have no choice we insert into the x-Fast structure and then into its respective group. Afterwards, insertion continues as normal where we check group(x) and then insert into the group, since we've only populated the x-Fast structure once all our group(x) queries less than the original insert value will take us to the same group. This is a problem as the subtrees can grow too large such that we lose the O(loglogn) properties. To fix this issue we limit the max size of a group to be 2 * logn. If the trees get too large we split them in half, find the maximum representitives of those groups and insert them into the x-Fast successor structure, making sure to move the tree's root pointers to their respective x-Fast successor leaf nodes. The way that we are able to do this is analogous to the array doubling problem, if we save up credits each time we do an insert that does not require splitting and re-indexing, by the time the tree gets too large we will have saved up logn many credits to make the operation "free". Ultimately having to only rarely insert into the x-Fast structure makes the complexity for insert $$O(loglogn)$$. The same case is analogous to delete. We only delete a subtree when it's empty. An important observation here is to note that even if we delete the maximum value within the subtree, we do not need to go into the x-Fast structure to delete the maximum representitive, the reason for this is because as the name suggests it is a representitive. Using this fact proving that delete is also O(loglogn) using amortization is a simple exercise.
In conclusion the structures outlined effectively take advantage of integer properties to speed up their operations. Shaving off another log factor for the successor problem should not be taken lightly. Given this remark; however, for Huffman Coding the viability of these methods should also be considered.
